# 企業核心 AI 功能與戰略分析

> **撰寫日期**：2026-02-19  
> **目的**：從企業實際痛點出發，系統性梳理企業最需要的 AI 功能模組，並評估 UniHR 的戰略定位與優先補強方向。

---

## 思考框架

企業的本質是「讓人協作完成目標」，所有 AI 需求都圍繞三個核心矛盾：

```
資訊太多  ↔  人力有限     → 需要「資訊消化」能力
知識分散  ↔  決策即時     → 需要「知識整合」能力
流程繁複  ↔  效率要求高   → 需要「流程自動化」能力
```

## 三、企業 AI 採購決策的關鍵考量

從採購端（CIO / CHRO 視角）來看，他們最在乎的不是「功能多不多」，而是：

| 採購關鍵因素 | 說明 | UniHR 現狀 |
|------------|------|-----------|
| **資料主權** | AI 不能把公司資料送到不明外部 | ✅ pgvector 本地向量；Voyage/OpenAI 僅傳送查詢文字 |
| **可解釋性** | AI 回答必須有來源，不能幻覺 | ✅ 來源引用機制（sources JSON）|
| **整合現有系統** | 能接 SAP / AD / SharePoint / Slack | ⚠️ SSO 已有；Webhook/API 待建 |
| **落地成本** | 不能比現有 SaaS 貴太多 | Free/Pro/Enterprise 三層定價 |
| **合規審計** | 誰問了什麼、AI 怎麼回答，要留紀錄 | ✅ AuditLog + UsageRecord |
| **漸進導入** | 不能 Big Bang，要能小範圍試驗 | ✅ Feature Flag + 租戶隔離 |
| **多區域 / 資料落地** | GDPR / 個資法合規 | ✅ region 欄位 (ap/us/eu/jp) |

## 四、對 UniHR 的三條戰略路徑

### 路徑 A：深耕 HR，做到極致

```
現有：知識問答（內規 + 勞動法）
加入：招募 AI（履歷篩選）
     → 績效評核輔助（KPI / 360 回饋）
     → 員工關懷預測（離職風險）
     → 教育訓練個人化（學習路徑推薦）
定位：HR 一站式 AI 工作台
競爭對手：Workday AI、SAP SuccessFactors AI、104 AI 招募
```

**優點**：深度護城河，不易被通用平台取代  
**缺點**：HR 買家決策週期長，市場規模受限

---

### 路徑 B：橫向延伸為「企業知識 AI 平台」⭐ 建議優先評估

```
現有：HR 知識問答（作為第一個客戶案例）
加入：法務合約問答
     → IT 知識庫問答
     → 財務政策問答
     → 任何部門的「內部文件問答」
定位：企業私有知識 AI 平台
核心差異化：多部門隔離 + 統一管理 + 一套計費 + 資料不出企業
競爭對手：Microsoft Copilot、Google Workspace AI、Notion AI
```

**優點**：
- 技術底層（RAG + 多租戶 + 安全隔離）**完全可複用**，不需要重建
- 每增加一個部門 = 增加一個租戶或知識庫，邊際成本極低
- 企業預算通常是「全公司採購」而非「單部門採購」，客單價 3-10 倍

**缺點**：競爭對手更強（Microsoft Copilot 在做同樣的事）

---

### 路徑 C：做平台底層，開放 ISV 整合

```
現有：RAG + 多租戶 + 安全隔離 + 配額管理
加入：Plugin SDK（ISV 可建垂直應用）
     → 開放 API（讓外部開發者建在上面）
     → Marketplace（應用商店）
定位：企業 AI 中台 / PaaS
競爭對手：Azure AI Foundry、AWS Bedrock
```

**優點**：平台效應，生態護城河  
**缺點**：需要大量開發者關係投入；技術複雜度高

**評估標準**：市場需求大 × 開發成本低 × UniHR 現有能力可複用

| 排名 | 功能 | 理由 | 複用 UniHR 現有能力 | 預估工時 |
|------|------|------|-------------------|---------|

## 七、合約審閱 AI 具體規劃（最高優先）

### 為什麼選這個

1. **文件解析已具備**：23 種格式，PDF/DOCX 是合約主要格式
2. **RAG + LLM 已具備**：換一個 System Prompt 即可
3. **多租戶隔離已具備**：不同公司合約完全隔離
4. **高客單價**：法務付費意願遠高於 HR
5. **快速上市**：3-5 天可做出 MVP

### MVP 功能範圍

```
Phase 1（3 天）：
  • 上傳合約文件（PDF/DOCX）
  • 自然語言問答（「這份合約的終止條款是？」）
  • 條款摘要（自動生成關鍵條款一覽表）

Phase 2（+3 天）：
  • 風險條款標記（依公司政策 / 法律常識）
  • 與範本比對（缺少哪些標準條款）
  • 多份合約比較

Phase 3（+5 天 → 正式產品）：
  • 合約版本管理（v1 vs v2 差異）
  • 審閱意見 / 批注協作
  • 匯出審閱報告（PDF）
```

### System Prompt 設計重點

```
合約審閱 AI 與 HR 問答 AI 的主要差異：
1. 角色：「你是合約審閱 AI，幫助法務人員快速理解合約內容並識別風險」
2. 重點：找高風險條款（無限賠償、自動續約、限制競業）
3. 格式：結構化輸出（條款名稱 + 原文 + 風險評估 + 建議）
4. 免責：「本 AI 分析僅供參考，重要合約請諮詢專業法律顧問」
```

## 八、會議紀錄 AI 具體規劃（第二優先）

### 輸入 / 輸出設計

```
輸入：
  • 音訊檔（mp3/mp4/wav）→ Whisper API 轉逐字稿
  • 文字逐字稿（直接貼入）
  • 影片連結（未來）

輸出：
  • 會議摘要（3-5 段）
  • 決議事項（Decisions）
  • Action Items（負責人 + 截止日期）
  • 待確認問題（Open Questions）
  • 出席者（若有提及姓名自動識別）
```

### 技術方案

```
音訊 → OpenAI Whisper API（轉錄）
      → 逐字稿作為「文件」上傳到現有文件系統
      → GPT-4o-mini 依結構化 prompt 輸出摘要
      → SSE 串流呈現（已有）
```

**複用現有元件**：
- 文件上傳 API（加 audio 類型支援）
- Celery 背景任務（加 Whisper 轉錄步驟）
- Chat + SSE 串流（完全複用）
- 用量追蹤（計費 Whisper 分鐘數）

## 十、總結

### UniHR 的核心競爭優勢（護城河）

1. **台灣勞動法規深度整合**：Core API + RAG 雙引擎，台灣本土 HR 問答準確率高於通用 AI
2. **企業級安全架構**：多租戶隔離、SSO、RBAC、稽核日誌 — 已超過大多數新創競品
3. **文件解析廣度**：23 種格式，含 LlamaParse 高品質解析，直接可擴充到其他部門

### 最重要的一句話

> **UniHR 的技術底層（RAG + 多租戶 + 安全隔離）已具備企業 AI 平台的骨架，最大的機會在於把「HR 知識問答」的 pipeline 複用到更多企業知識場景（法務、IT、財務），而非繼續在 HR 單一功能上深挖。**
